## force_kill 实现排查（发现现有 bug）

- `publisher.py:251` 调 `self._force_kill_service_on_worker()`，但**该方法全项目零定义**——watchdog 超时分支一旦触发立即 `AttributeError`，救援路径从未真正工作过。
- `BinanceService`（隔壁 `../binance-service` 包，老李自己的项目）只有 `open()` / `close()`，**没有 `force_kill` 方法**；测试里的 `force_kill` 是 mock。
- `close()` 也救不了：走 `context.close()` / `browser.close()`，走同一条 CDP 通道，CDP 一卡 `close()` 也卡。
- Chrome 起源：`_playwright.py` 里 `pw.chromium.launch()` 由 Playwright fork，PID 在 Playwright 内部对象 `browser._impl_obj._channel._connection._transport._proc` 上，无公开 API。
- 三条实现路径已给老李拍板：
  - 路径 A（推荐）：在 publisher.py 实现 `_force_kill_service_on_worker`，走 Playwright 私有属性拿 PID + `proc.kill()`。精准、不动隔壁包。风险：Playwright 大版本升级内部属性可能坏。
  - 路径 B：psutil 按 user-data-dir 过滤杀 Chrome。需加依赖、跨进程扫。
  - 路径 C：改 binance-service 的 `_playwright.py`，自己 `subprocess.Popen` 启 Chrome + `connect_over_cdp` 连接。架构最干净但改动大。
- 重要纠正：老李原想"杀整个 worker 进程"——不行，worker 是 trading-service 主进程的线程，杀进程等于把整个服务干掉。**杀 Chrome 进程就够了**，目标是解 worker 阻塞。等老李拍板路径。

## 老李拍板：超时就重启整个服务（不做路径 A/B/C）

- 老李新方案：watchdog 超时 → `os._exit(1)` 让进程退出 → systemd 自动拉起重启。比拿 Playwright 私有属性拿 PID 简单 10 倍。
- 部署层已就绪：`deploy/deploy.sh` 用 systemd，已配 `Restart=always` + `RestartSec=10`，进程退出后 10s 自动拉起。
- 配套措施 3 项：
  - 配套 1（防重启循环）：deploy.sh 加 `StartLimitIntervalSec=300` + `StartLimitBurst=3`。5 分钟内重启超 3 次则停止重启，等人工介入。
  - 配套 2（清残留 Chrome）：启动时 `pkill -f "remote-debugging-port=<port>"`。os._exit 是硬终止，Chrome 可能成孤儿，下次启动撞端口。老李说"以后再说"。
  - 配套 3（队列 dump）：重启后队列剩余任务会丢。老李说"以后再说"。
- 为何用 `os._exit(1)` 而非 `sys.exit(1)`：`sys.exit` 抛 `SystemExit`，主线程能接住但 worker 卡在 CDP 通道不响应；`os._exit` 立即终止进程绕过所有清理，粗暴但可靠。副作用：日志 buffer 末尾几行可能丢，streaming handler 一般即时 flush 影响可忽略。
